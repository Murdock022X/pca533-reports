\section{Cornerstone}
\label{sec:cornerstone}

To characterize Cornerstone's performance we first need to understand the algorithmic steps. Note that there are two methods of interest, the method where an octree is constructed locally on a node and the case where a locally essential tree is used for distributed simulation. We review the local case first and then advance to the distributed case as an extension. We focus primarily on the code needed and how it maps to GPU architectures, for further insight to the algorithm reference Cornerstone ~\cite{cornerstone}. An outline of the steps in construction is given for the local case.
\begin{itemize}
    \item Compute a space filling curve (Morton or Hilbert)
    \item Sort the keys and rearrange any properties associated with the key such as the position, mass, charge, etc
    \item Update the leaf nodes
    \item Update the internal tree structure
\end{itemize}

Additionally Cornerstone has 6 critical parameters which are tabulated below to give an overview. These will also be important parameters for characterization.

\subsection{Space Filling Curves}

Informally a space filling curve will take points existing in space and map them to integer keys, when one sorts the original particles based on these keys and traverses the path through them the resulting curve will recursively move through the domain octant by octant (in the case of three dimensional points). In this way a space filling curve is a kind of fractal.

We are interested in two types of space filling curves namely Morton Z-order curves and Hilbert curves. The path of a Morton Z-order curve may be viewed in figure \ref{fig:morton_leaves} and shows the distinct z-pattern. Morton codes are computationally efficient to calculate, requiring just a few bit shifts, but as is also evident visually in figure \ref{fig:morton_leaves} the curve may "jump" octants this may result in unfavorable communication patterns between nodes.

The Hilbert curve on the other hand takes on a compact U shape as seen in figure \ref{fig:hilbert_leaves}. Hilbert codes offer a "compact" mapping which may reduce the cost of communication but come at the cost in increased computational inefficiency requiring $L$ loop iterations, $L$ being the maximal depth of the octree.

In both cases the calculation is a simple elementwise GPU kernel, which does a relatively small number of operations.

\subsection{Radix Sorting and Gathers}
We will not detail the Radix sort and Gather implementations as they are well known. The reason these operations are necessary is to sort the keys thus producing the in-order arrangement of keys according to the space filling curve that recursively partitions the space into octants. The keys will have some attributes including $x,y,z$ but may have others based on the application, these also need to be aligned with the sorted keys. In Cornerstone these are distinct kernels with a Radix sort and $k$ gathers for the number of properties to be sorted. The radix sort is done using the CUB libraries implementation DeviceRadixSort::SortPairs. The gathering of particles in the single node case is left to the user but we used the Thrust libraries gather implementation which has competitive performance.

\subsection{Update Leaf Nodes}
The namesake of Cornerstone ~\cite{cornerstone} is the Cornerstone array called $K$. $K$ is a sorted array of SFC keys of length $n_l+1$, $n_l$ being the number of leaves, with the first element being $0$ and the last element being $8^L$, with the difference between one element and it's successor being $8^l$, where $l \le L$. Then $K$ will define a unique octree but represents explicitly the leaves. Cornerstone ~\cite{cornerstone} also uses an $N_{crit}$ parameter which defines the maximum allowable number of particles a leaf node may have; in other words this parameter defines the resolution of the octree or depth at which it must be resolved. Then the steps to generate the Cornerstone array from the particle SFCs are to calculate a histogram of the particle SFCs belonging to each each leaf node and then perform three steps:
\begin{enumerate}
    \item Compute the rebalancing decision for each node i.e a code where 8 indicates $N_{crit} < N_i$ and so the node should be subdivided, 0 indicates the node should be replaced, and 1 indicates the leaf should be unchanged.
    \item Perform an exclusive prefix sum of the array of computed rebalancing decision codes, this leads to each element in the result being the new index of the node in the corresponding new Cornerstone array. The final element is the number of nodes in the new Cornerstone array.
    \item Finally, if the rebalancing decision for node $i$ is $1$ then the key is copied to the new index calculated in the exclusive prefix sum. If the decision is to split, the code is $8$ and we copy the keys of the children of the original node to the new range.
\end{enumerate}
A Cornerstone array is then balanced once no node needs to be split or merged, which means that every leaf node has a particle count $\le N_{crit}$ and every internal node has a particle count $> N_{crit}$. By iteratively applying these steps until we reach the criteria for a balanced Cornerstone array we define a unique balanced octree. 

Regarding performance, note that each kernel launch here must occur in serial, no stream parallelism is possible as global synchronization is necessary between each step, however there are some techniques that may allow kernel fusion to be effective which may be of interest depending on the high level bottlenecks. The calculation of the rebalancing decision is an elementwise kernel and in practice may use larger multiples of $8$ to indicate multiple levels need to be merged. The exclusive prefix sum Additionally, a memory allocation is necessary and has to read the value of the last element of the exclusive prefix sum to calculate the number of new leaf nodes. 

\subsection{Update Internal Nodes}
Finally the internal tree structure must be created allowing us to traverse the tree and ultimately we are left with three arrays of integers, note $n = n_i + n_l$ with $n_i$ being the number of internal nodes and $n_l$ being the number of leaf nodes:
\begin{enumerate}
    \item $NK$ of size $n$, with each value being within the range defined by the SFC size supported by $L$ which is $[0,2^{3L})$. This is the SFC key for each node both leaves and internal nodes, note that because the SFC can decode to 3D points this also tells us the 3D positions of each node.
    \item $CO$ of size $n$, with each value being in the range $[1,n)$. This is the connectivity array.
    \item $LO$ of size $L + 2$ with each value being in the range $[1,n)$ which are the tree level offsets, that is the $j$ element of $LO$ gives the index into $CO$ and $NK$ that is the first node in the level $j$ of the tree.
\end{enumerate}

This requires several operations, the first is allocating $NK$ and copying the Cornerstone array to the portion of the range for leaf nodes, this is an elementwise kernel. Then the algorithm computes another elementwise kernel that does an xor of two adjacent keys in the Cornerstone array and then counts the leading zeros. Then $NK$ is sorted via radix sort which gives a breadth-first node ordering, again using the CUB implementations. Then the level offsets $LO$ are generated with a binary search of $NK$, another elementwise kernel. And finally the connectivity array $CO$ is generated by doing a binary search for the first child of a node using the level offsets to reduce the problem size.

An important performance note here that will show up in \ref{sec:results} is that the computation of SFC keys and the subsequent radix sort and gathers all operate on $n_p$ which is the number of bodies in simulation space which is much larger than the computation being done in the later two steps where we operate only on the leaf nodes assuming $N_{crit}$ is sufficiently large. We discuss in further detail in \ref{sec:results}.

\subsection{Domain Decomposition}

The above process outlines the process to build a balanced octree on a single rank. This method can be extended however to allow multiple peers to decompose the domain among themselves, this allows a rank to maintain a well resolved local domain with a coarser global domain. This modifies the process of generating the octree to compute a global octree and a focused tree. 

The global octree allows us to decompose the domain of particles among the peer ranks based on the SFC, calculating the coarse global domain with a higher $N_{crit}$ called $N_{crit,glob}$. This process modifies the above algorithm so that the initial histogram step in the Cornerstone array rebalance does a global sum of the histograms from each rank via \textit{MPI\_Allreduce}. Then performing the balancing steps using the histogram of the global domain rather than the local domain yields a balanced global octree. Such an algorithm would incur unacceptable communication costs at high resolution (lower $N_{crit}$) as it involves an all-to-all communication which would be iterated on; more resolved domains requiring more iterations to balance the tree. However at a coarser resolution we can use this to efficiently decompose the global domain by partitioning the Cornerstone arrays space filling curve among the ranks the collective communication of this method is $O(nlogn)$ where $n$ is the total number of particles across all ranks.

\subsection{Locally Essential Trees}

Once the space filling curve is decomposed among $n_{ranks}$ subdomains the algorithm attempts to find a Locally Essential Tree which includes the nodes necessary for computation within the local subdomain $F$. Here Cornerstone ~\cite{cornerstone} incorporates physical context into the algorithm considering both short-range forces and long-range forces. Both require a high resolution with $F$ and at the exterior surface of $F$ for halo particle discovery. Long-range forces are assumed to be computed via Barnes-Hut or the Fast Multipole Method both of which use a Multipole Acceptance Criterion (MAC) function to determine whether the multipole approximation applies given a source node and target point or node; the multipole approximation being that the force can be computed by using the source node as a kind of super particle that acts on the target with a force relative to the number of particles within it. The most common MAC, where $r_{min}$ is the minimum distance between source and target, $l$ is the source node edge length, and $\theta$ is a controllable accuracy parameter as noted in the table above, is of the form $r_{min} > l/\theta$, if this inequality is true then the the multipole approximation is valid. Note that as the $\theta$ parameter is increased the multipole approximation is accepted at smaller distances or for larger source node lengths, this tends to make the solution less accurate as the approximation usually becomes less valid for nodes that are larger or closer to the target. 

Finally this results in several new arrays and functions to balance this Locally Essential Tree. We also represent a LET with a Cornerstone array and construction remains similar with some tweaks. First we compute whether whether the multipole approximation is accepted for every node outside of $F$, the multipole approximation is marked as not acceptable if it fails for any target node in $F$, the array of evaluations is called $M$. Some MACs may require communication between ranks. Rebalancing is now additionally a function of $M$ and does not require communication while generating the histogram of node counts requires communication. The bottleneck according to Keller et al. ~\cite{cornerstone} is the function calculates the node properties based on particles and the nodes children which is done via an upward pass using point-to-point communication for particles outside of $F$ but where the LET has finer resolution than the global tree (category 2) and collective communication for particles outside of $F$ in nodes where the resolution of the global tree exceeds that of the LET; note that the LET has a variable resolution outside of $F$ based on the MAC. The overall process is as follows then:
\begin{enumerate}
    \item First the node properties for leaves in $F$ are computed locally (category 1). Each rank finds category 2 nodes by traversing the global octree and evaluating as stated above. 
    \item Ranks send p2p messages of their octree nodes to peer ranks. The receiver uses the nodes to fill in properties for category 2 nodes and then the node properties are sent back to the requesting peer.
    \item Perform an \textit{MPI\_Allgatherv} communication to all nodes to send the aggregated LET properties across the ranks, then perform another upward pass on each node to get the properties all the way up to the global root node.
\end{enumerate}

This yields a focused LET at each rank allowing traversal of the local subdomain and the calculation of forces.
